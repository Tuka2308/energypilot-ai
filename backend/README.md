# EnergyPilot AI — backend

FastAPI-бэкенд под MVP-скоуп (см. корневой `CLAUDE.md`). Все четыре фичи —
распознавание счёта, прогноз, аномалии, AI-чат — уже с реальной логикой
(`app/services/*`).

## Как строится прогноз

Одна модель — **Prophet** (по MVP-скоупу: без ансамбля LSTM/XGBoost).
Вход — история подтверждённых счетов профиля (по точке на месяц), которую
`bill_history_service` накапливает из bills-эндпоинта (upload с `profile_id`
или ручная правка). Prophet моделирует тренд и из коробки отдаёт
доверительный интервал (`yhat_lower/upper`) — показываем его как честный
разброс, а не одну цифру.

- **< 3 месяцев истории** → статус `insufficient_history` без числа: у нового
  пользователя Prophet дал бы неустойчивый прогноз, поэтому не форсируем.
- **3–23 месяца** → только тренд: годовая сезонность на < 2 годах данных
  катастрофически переобучается (проверено — прогноз улетал в сотни тысяч
  тенге), поэтому включается лишь с ~24 месяцев.
- Прогноз строится на месяц после последнего в истории (текущий незакрытый
  счёт). Потребление в кВт·ч оценивается через средний тариф по истории.

## Аномалии

Пороговое правило (`app/services/anomalies_service.py`), не ML — по
MVP-скоупу. Сравниваем потребление последнего месяца с базой пользователя;
рост **≥ 30%** флагуем как аномалию. Порог не произвольный: помесячный
бытовой разброс без смены привычек ~±10-20%, +30% — это ~1,5-2 таких
отклонения, то есть ловим поведенчески значимый рост рано, не заваливая
ложными срабатываниями. Серьёзность градируется (≥60% → high). Флагуем
только рост (боль — большой счёт, не падение). База сравнения — тот же
месяц год назад, если он есть в истории (учёт сезона), иначе среднее.
Метрика — кВт·ч (поведение), при отсутствии — сумма. При < 3 месяцев
истории → `insufficient_history`, как и у прогноза. Ответ содержит
человекочитаемое объяснение и структурированные поля (метрика, значения,
база) — под карточку UI и промпт AI-чата.

## AI-чат энергокоуч (structured pipeline)

`app/services/chat_service.py` — НЕ обёртка над chat API. Пайплайн:

1. **Сбор контекста** — по `profile_id` из тех же сервисов, что кормят
   дашборд: профиль/техника/тариф (онбординг), прогноз Prophet с
   интервалом, аномалии со структурированными полями. У чата нет своей
   аналитики — его цифры не могут разойтись с дашбордом.
2. **Блок фактов** — контекст рендерится в секции КВАРТИРА/ТЕХНИКА/ТАРИФ/
   ПРОГНОЗ/АНОМАЛИИ. При `insufficient_history` в блоке стоит явный запрет
   называть сумму прогноза — защита от галлюцинаций, а не молчаливый
   пропуск секции.
3. **Правила** — системный промпт требует советы только из фактов, каждую
   рекомендацию с оценкой экономии и показанным расчётом; ответ — строгий
   JSON `{reply, estimated_savings_tenge}`.
4. **Память диалога** — история сообщений in-memory по `profile_id`
   (последние 8 пар), уточняющие вопросы работают в рамках сессии.
5. **Каскад LLM** — первый настроенный и рабочий провайдер побеждает:
   **OpenAI** (JSON-mode, `{reply, estimated_savings_tenge}`, рекомендованный
   путь) → Gemini (`gemini-2.0-flash`, тот же JSON-контракт) → Ollama
   офлайн (через stdlib) → детерминированный fallback из собранного
   контекста с пометкой офлайн-режима. Сбой одного провайдера не роняет
   эндпоинт — пробуем следующий по каскаду.

   **Почему OpenAI первым, а не Gemini:** у команды бесплатный ключ Gemini
   оказался заблокирован на уровне аккаунта — `429 RESOURCE_EXHAUSTED`,
   `limit: 0` для free tier `gemini-2.0-flash`. Проверено сырым `curl`
   напрямую к `generativelanguage.googleapis.com` в обход этого кода —
   тот же результат, и он не меняется при повторной попытке (значит не
   временный burst-лимит, а структурное ограничение: обычно означает, что
   Google-аккаунт/проект, на который выпущен ключ, не подходит под free
   tier в этом регионе — например, аккаунт организации/Workspace вместо
   личного Gmail). Это не баг интеграции: `_call_gemini` и парсинг ответа
   написаны и остаются в каскаде — заработают сами, если квоту почините
   (пересоздать ключ на другом аккаунте/проекте в
   https://aistudio.google.com/apikey).

Включение LLM — заполнить один из ключей в `backend/.env`:

| Переменная | Провайдер | Где получить |
|---|---|---|
| `OPENAI_API_KEY` | **Рекомендовано.** OpenAI (`OPENAI_MODEL`, по умолчанию gpt-4o-mini) | platform.openai.com |
| `GEMINI_API_KEY` | Google Gemini (`gemini-2.0-flash`) — см. оговорку выше про региональное ограничение | https://aistudio.google.com/apikey |
| `OLLAMA_BASE_URL` | Локальная модель (`OLLAMA_MODEL`) | см. таблицу ниже про Docker |

Без ни одного из них чат работает в честном офлайн-режиме — демо не
зависит от сети/ключей.

**Ollama + Docker:** `OLLAMA_BASE_URL` подставляется в запрос как есть, без
подмены (см. `_call_ollama` в `chat_service.py`) — значение должно быть
адресом, реально доступным из места, где выполняется код:

| Backend запущен | Ollama на хосте | Значение OLLAMA_BASE_URL |
|---|---|---|
| напрямую (`uvicorn`, вне Docker) | Mac/хост | `http://localhost:11434` |
| в `docker-compose` | Mac/хост | `http://host.docker.internal:11434` |

Внутри контейнера `localhost` — это сам контейнер, а не хост-машина, где
реально слушает Ollama. Ошибка при этом **не громкая**: `_call_llm` ловит
исключение подключения и молча уходит в офлайн-fallback, поэтому чат
отвечает «LLM не настроен», даже если `OLLAMA_BASE_URL` заполнен — первый
симптом бага именно такой. `host.docker.internal` резолвится «из коробки»
на Docker Desktop (Mac/Windows); `docker-compose.yml` также явно прописывает
`extra_hosts: host.docker.internal:host-gateway` для backend-сервиса — это
нужно на Linux (Docker ≥ 20.10), где автоматического резолвинга нет.

## Как распознаётся счёт

Два пути по типу входа:

1. **Цифровой PDF** (типичная "Ведомость потребления электроэнергии" ЭСО) —
   это таблица с текстовым слоем, а не картинка. Читаем её структурно через
   `page.find_tables()` PyMuPDF и берём значения из конкретных колонок по
   заголовкам ("Сумма с НДС" → сумма к оплате, "Потреблено" → кВт·ч). OCR
   здесь НЕ применяется: он превращал бы точный текст в шум и путался в
   плотной таблице (из-за чего сервис однажды уверенно вернул год "2026"
   как сумму).
2. **Фото счётчика / сканированный PDF** без текстового слоя — здесь OCR
   (Tesseract) по ключевым словам рядом с числом.

Доверие проверяется не только диапазоном значений: в табличном пути сумма
принимается, только если непротиворечиво читается из своей колонки (итог с
НДС ≥ суммы без НДС); в OCR-пути нет жадного фолбэка "первое число в
диапазоне". Любая неоднозначность/низкая уверенность → `requires_manual_review:
true`, а не тихо-неверное число.

## Структура

```
app/
  main.py            # создание FastAPI-приложения, CORS, регистрация роутеров
  core/config.py      # настройки из env/.env
  routers/            # HTTP-слой: онбординг, счета, прогноз, аномалии, чат
  models/schemas.py   # Pydantic-контракты запросов/ответов
  services/           # бизнес-логика: bills_service.py — OCR/разбор счёта,
                       # bill_history_service.py — история счетов профиля,
                       # forecast_service.py — прогноз (Prophet),
                       # аномалии/чат пока моки
```

## Системные зависимости

OCR загрузки счёта использует `tesseract-ocr` через `pytesseract` — сам
бинарник pip не ставит, нужен отдельно:

```bash
# macOS
brew install tesseract
# для распознавания кириллицы (казахстанские счета на русском) —
# положить rus.traineddata в tessdata, brew ставит только eng/osd/snum:
curl -fsSL -o "$(brew --prefix tesseract)/share/tessdata/rus.traineddata" \
  https://github.com/tesseract-ocr/tessdata_fast/raw/main/rus.traineddata

# Debian/Ubuntu
sudo apt-get install tesseract-ocr tesseract-ocr-rus
```

Прогноз использует **Prophet**, который считает через CmdStan (C++).
`cmdstanpy` компилирует его при первом использовании — локально это
разовый прогрев (`python -c "import cmdstanpy; cmdstanpy.install_cmdstan()"`,
~4 мин), в Docker он собирается на этапе build (см. `Dockerfile`,
`build-essential` + `install_cmdstan`), чтобы первый запрос к `/forecast`
не висел минуты.

В Docker-образе (`Dockerfile`) tesseract и CmdStan уже прописаны.

**Честно про качество:** OCR-движок офлайн и бесплатный, но заметно слабее
облачных на неровных/тёмных фото — таков компромисс за отсутствие
ключей/оплаты на хакатоне. Табличный путь для цифровых PDF, наоборот,
точный (читаем текстовый слой). В обоих случаях при сомнении сервис не
подставляет случайные цифры, а возвращает `requires_manual_review: true`.

## Локальный запуск

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
uvicorn app.main:app --reload
```

Открыть `http://localhost:8000/docs` — Swagger со всеми эндпоинтами.

Через Docker — см. `docker-compose.yml` в корне репозитория.

## Тесты

```bash
pytest
```

- `tests/test_bills_ocr.py` — регрессия распознавания счёта, в т.ч. на
  синтетической квитанции ЭСО (`tests/fixtures/sample_receipt_kz.pdf`):
  сумма/потребление/период извлекаются верно, мусорный вход уходит в ручную
  проверку, а не в 500.
- `tests/test_forecast.py` — прогноз: нехватка истории → `insufficient_history`
  без числа; на истории за несколько месяцев прогноз в разумном диапазоне и
  внутри своего доверительного интервала (ловит прежнее переобучение
  сезонности). Тест с Prophet обучает модель (~1 c, требует CmdStan).
- `tests/test_anomalies.py` — пороговое правило аномалий: нормальный рост не
  флагуется, скачок ≥ порога флагуется с объяснением, нехватка истории →
  `insufficient_history`; плюс год-к-году сравнение и фолбэк на сумму.
- `tests/test_chat.py` — чат-пайплайн (LLM не вызывается, ключи обнулены):
  блок фактов содержит профиль/прогноз/аномалию, при нехватке истории —
  явный запрет вместо числа; офлайн-fallback осмыслен; память диалога
  хранит пары по профилю.
